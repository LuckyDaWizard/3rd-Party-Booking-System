<Section id="tls-hsts" num="03 — TLS + HSTS" title="TLS termination and HSTS">

**TLS** is terminated at Traefik. Configuration lives entirely in router labels on the `booking-app` service:

```yaml
- "traefik.enable=true"
- "traefik.http.routers.booking.rule=Host(`bookings.carefirst.co.za`)"
- "traefik.http.routers.booking.entrypoints=websecure"
- "traefik.http.routers.booking.tls=true"
- "traefik.http.routers.booking.tls.certresolver=letsencrypt"
- "traefik.http.services.booking.loadbalancer.server.port=3000"
```

The `letsencrypt` cert resolver is defined on the Traefik container itself (HTTP-01 ACME challenge, auto-renewed). ACME storage lives in a named Docker volume so renewals survive container restarts.

**HSTS** is enforced via a Traefik headers middleware, also defined via labels:

```yaml
- "traefik.http.middlewares.hsts.headers.stsSeconds=63072000"
- "traefik.http.middlewares.hsts.headers.stsIncludeSubdomains=true"
- "traefik.http.middlewares.hsts.headers.stsPreload=false"
- "traefik.http.routers.booking.middlewares=hsts"
```

`max-age=63072000` is 2 years. `includeSubDomains=true` covers any subdomain we add later (`api.bookings.carefirst.co.za`, etc.). `preload=false` is deliberate until the soak window passes.

**HTTP → HTTPS redirect** is configured at the Traefik entrypoint level, not as a per-router label:

```
--entrypoints.web.address=:80
--entrypoints.websecure.address=:443
--entrypoints.web.http.redirections.entrypoint.to=websecure
--entrypoints.web.http.redirections.entrypoint.scheme=https
```

Verify with `curl -sI https://bookings.carefirst.co.za` — expect `strict-transport-security: max-age=63072000; includeSubDomains` in the response.

</Section>

<Section id="deploy" num="04 — Deploy" title="Deploy and rollback">

Deploys are SSH-only. Image gets tagged with the commit SHA on every deploy so the previous image stays available for instant rollback.

```bash
ssh root@187.127.135.11
cd /opt/3rd-Party-Booking-System
git pull origin main
export IMAGE_TAG=$(git rev-parse --short HEAD)
cd booking-app
docker compose build
docker compose up -d
```

<Callout variant="warn" title="docker-compose.yml drift between repo and VPS">
The repo's `booking-app/docker-compose.yml` does **not** currently include the production Traefik labels (TLS resolver, Host rule, HSTS). The live VPS file has them; the git file doesn't. Until the drift is resolved (backlog B4), the simple `git pull` step above is safe **only because** the live compose has been edited via SSH and no commit on `main` has rewritten the compose file. If you change the repo's compose, audit the live file first or you'll wipe the production labels.

Inspect the live compose before doing anything risky:

```bash
cat /opt/3rd-Party-Booking-System/booking-app/docker-compose.yml | grep traefik
# Expect 10 traefik labels: 6 router + 4 HSTS middleware
```
</Callout>

Rollback is a single command — no rebuild, ~10 seconds:

```bash
export IMAGE_TAG=<previous-short-sha>
docker compose up -d --no-build
```

Health check post-deploy:

```bash
curl -sf https://bookings.carefirst.co.za/api/health
# → { "status": "ok", "time": "...", "checks": { "db": "ok" } }
```

</Section>

<Section id="tests" num="05 — Tests" title="Automated tests (Playwright)">

First end-to-end regression net shipped 2026-06-08 (backlog B3). Chromium-only; idempotent service-role seed gated on `PLAYWRIGHT_SEED=1`.

```bash
cd booking-app

# First time on a fresh dev DB — seeds Playwright Test Clinic / Unit / User / Coupon
PLAYWRIGHT_SEED=1 npx playwright test

# Every time after — seed survives, no DB writes from setup
npx playwright test

# HTML report (after a failure)
npx playwright show-report
```

Current coverage:

<Section id="caveats" num="06 — Caveats" title="Operational caveats">

<Callout variant="warn" title="Don't touch the Hostinger Docker Manager UI for compose edits">
Saving anything via Hostinger's hPanel compose-edit form (including env vars) regenerates `docker-compose.yml` and wipes the manual Traefik labels. The team has been bitten twice during HTTPS rollout. Always SSH + nano; never click Save in the Docker Manager panel after manual edits. Env vars can be set via the Hostinger Environment panel (which doesn't touch the compose file).
</Callout>

<Callout variant="warn" title="FortiGuard block at the First Care Solutions office">
The office FortiGate categorises the raw IP `187.127.135.11` as "Unrated" and blocks it. The `bookings.carefirst.co.za` subdomain is categorised as Business and is unblocked. Always smoke-test against the subdomain, not the IP. Corporate MDM may push the same filtering to managed phones even on cellular.
</Callout>

<Callout title="1 vCPU constraints">
The Hostinger KVM 1 plan is 1 vCPU. Any `docker-compose cpus:` value must be `< 1.0`. The current setting is `cpus: 0.9` which leaves a sliver for Traefik + the host kernel. Bumping to ≥ 1.0 causes Docker to refuse with `range of CPUs is from 0.01 to 1.00`. The reconcile cron uses a 5-in-flight concurrency pool for the same reason (Sprint A C6).
</Callout>

</Section>

<Section id="implications" num="07 — Implications" title="Implications for CareFirst">

<Callout title="Inbound CareFirst webhooks are unblocked">
HTTPS + HSTS means we can now safely accept signed webhooks from your team — for example a consultation-outcome webhook that tells us when a session completed, was a no-show, etc. The right sequence:

<ol>
<li>You build the webhook against <code>https://bookings.carefirst.co.za</code> with shared-secret + signature validation</li>
<li>We add the receiving endpoint, validator, and the corresponding booking state transition</li>
<li>We deploy in one direction and you fire test traffic before going live</li>
</ol>

This was the gating dependency in the consultation-outcome webhook RFC — it's now cleared.
</Callout>

<Callout title="One-way HTTPS for outbound calls — unchanged">
Outbound HTTPS calls (SSO auto-register and any future endpoints you expose) have always worked. Nothing changes here.
</Callout>
