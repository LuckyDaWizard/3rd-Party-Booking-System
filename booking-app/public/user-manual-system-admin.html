User Manual
Role: System Admin

CareFirst Third Party Booking System

Full operational and administrative guide.
Version 1 · Last updated 2026-06-08

You are a System Admin. You have the most powerful role in the system — complete access to every client, unit, user, audit-log entry, and security control. This manual covers everything that is exclusively yours. For day-to-day booking flow and unit manager tasks, refer to the User and Unit Manager manuals.

Power & responsibility Every action you take is logged in the Audit Log against your name. Multiple System Admins provide oversight of each other — there is no “super-admin” that hides activity. Treat this access accordingly.

1. What you can do

Every UI page and every API endpoint is open to you. In practice, the admin-only areas are:

AreaCapabilities
ClientsCreate, edit, and delete clients. Set branding, payment mode, and feature switches.
UnitsSame as a Unit Manager, but across all units, not just assigned ones. You can re-parent a unit to a different client.
UsersCreate users of any role (User, Unit Manager, System Admin), assign them to any unit, reset PINs, toggle status.
Audit LogView every entity create / update / delete / reset_pin / toggle_status action across the system, filterable by actor, date, entity type.
SecurityFailed sign-in attempts, active sessions (with revoke), suspicious activity, sign-in history.
OperationsTrigger PayFast reconciliation and run cleanup sweeps (most of this is automated by the server’s 15-minute schedule).
PrivacyProcess POPIA right-to-erasure requests via the privacy API.

Data visibility scope — you see everything. Unlike Users and Unit Managers (whose Home / Patient History / Create-Booking search are restricted to units they’re assigned to), you see every booking across every unit and every client. Patient History and the Create-Booking search will return matches from the entire system. This is handy for support and debugging, but it means your own testing does NOT reflect what an operator would see — if you want to verify unit / client isolation is working, sign in as a User-role account scoped to a single unit.

3. The data hierarchy

It helps to keep the hierarchy clear in your head before navigating the admin pages:

  • Client — the company / brand. Holds branding, payment mode, feature switches.
  • Unit (belongs to one client) — a physical clinic location. Inherits branding and payment mode from its client.
  • User (can be assigned to many units) — a person. Their effective “active” unit determines what they see.
  • Booking (belongs to one unit) — one patient visit.

This means client-level changes (e.g. branding) cascade to every unit; unit-level changes only affect that unit; user-level changes only affect that user.

4. Client Management

Click Client Management in the sidebar. You see a list of every client with their logo, name, contact, and unit/user counts. Click any row to manage that client, or click + Add Client to create a new one.

5. Adding a new client

The Add Client wizard walks you through 4 steps:

  1. Client Details
    Client name, contact person, contact email, contact phone. Used on receipts and emails.
  2. Branding
    Upload logo (appears in the sidebar when a user is on one of this client’s units), favicon (browser tab icon), accent colour (used on buttons and highlights). Defaults are fine to start with — you can always edit later.
  3. Settings
    Choose payment mode and feature switches (see §7 below).
  4. First Unit
    Every client needs at least one unit to be useful. Create one now with name, code, and address; you can add more later from the Units tab.

6. Managing an existing client — tabs

Click any client row to open it. The page has five tabs:

Client Details (editable)
Client name, contact person, email, phone.
Branding (editable)
Logo, favicon, accent colour. Changes are immediate — users on this client’s units will see the new branding on their next page load.
Settings (editable, admin-only)
Payment-mode and feature switches. See section 7 for full descriptions.
Units (read-only list)
Every unit belonging to this client. Click a unit name to jump to its Unit Management page. Add new units there, not here.
Users (read-only list)
Every user assigned to any unit under this client. Includes their role, status, and last sign-in. Read-only here — edit users from User Management.

7. Client Settings switches

The Settings tab on a client is where the client’s personality is configured. Each switch cascades to every unit and every booking made at this client.

SettingWhat it controls
Payment mode PayFast: the patient pays online during booking.
Self-collect: the clinic collects payment at the counter; the system just records it.
Monthly invoice: First Care Solutions invoices the client monthly; no money collected at booking time.
Self-collect at client level Master switch that enables “self-collect” mode for every unit of this client. Without this flag, the unit can’t opt into self-collect.
Skip patient metrics When ON, the Patient Metrics step is hidden from the booking flow. Useful for clinics that don’t capture vitals at intake.
Nurse verification When ON, the booking flow shows PIN prompts at the start (patient search) and the end (Start Consult) as a second-factor checkpoint. Any Active user assigned to the unit — including regular Users — can authorise their own booking; you don’t need a separate manager to be present. Default OFF.
Allow coupons When ON, the PayFast payment step shows an “Apply coupon” input. Patients (or operators on their behalf) can apply a discount code created in Coupon Management (see §8 below). Only works on the PayFast gateway path — the switch is force-disabled when the client is on self-collect or monthly-invoice. Default OFF.
Effect is immediate Changing payment mode while a user is mid-booking can confuse them. Prefer to change settings during quiet hours.

8. Coupon Management

Coupons are discount codes that patients (or operators on their behalf) can apply on the PayFast payment step. System Admin only. Find it in the sidebar under Coupons.

The list page

Every coupon in the system is shown here, across all clients. Each row carries the code, discount, scope (Global or one client’s favicon), validity window, usage counter, and a Manage button.

Adding a coupon

Click Add Coupon. The form mirrors the Add User / Add Client wizards in shape:

  • Code — case-insensitive. Stored uppercase.
  • Discount type + value — Percentage (1–100) or Fixed (rand amount).
  • Scope — Global (works for any client) or scope to a single client. Scoped coupons reject on apply if the booking’s unit belongs to a different client.
  • Validity window — optional “valid from” / “valid to” dates.
  • Usage limit — total applies across all patients (optional).
  • Usage limit per email — how many times one patient (by email) can use this code (optional).
  • Min spend — minimum consultation fee for the code to apply (optional).
Releasing usage on Abandon / Discard When a booking with a coupon is moved to Abandoned (idle timeout) or Discarded (user action), the system automatically frees that usage slot — the coupon’s counter decrements and the code can be applied again. This is enforced by a database trigger, not application code, so it works even when the booking is discarded from the database directly.

The R0 case (full comp)

If a coupon takes a booking to R0 — a 100%-off code or a fixed amount worth more than the consultation fee — PayFast is skipped entirely. The patient sees a green Complete Free Booking button instead of Pay With PayFast. The resulting booking has payment_type = "coupon_comp" and an emerald “Comped” chip on Patient History so it can be distinguished from real gateway payments in reports.

Managing an existing coupon

Click Manage on a row. You can edit any field, see how many times the code has been used so far, and (when usage is zero) hard-delete it. Once a coupon has been used at least once, deletion is blocked — archive it by setting valid to in the past instead.

Reporting

The Patient History CSV export includes Coupon Code, Original Amount, Discount Amount, and Final Amount columns next to Payment Type, so coupon usage can be reconciled against consultation revenue.

9. Unit Management (admin view)

Unit Management for an admin is the same as for a manager — except you see every unit in the system, not just your own. The only additional capability:

  • You can re-parent a unit to a different client. This is a heavy action: it cascades through bookings, users, and audit log. Do it deliberately.
Deletion Deleting a unit also removes its bookings permanently. Prefer to deactivate (toggle status) rather than delete unless you have a clear reason.

10. User Management (admin view)

Same page as the manager view, but with the full role list (User, Unit Manager, System Admin) available when adding or editing. Specific admin-only powers:

  • Create other System Admin and Unit Manager accounts.
  • Assign a user to any unit (manager is limited to their own units).
  • Change an existing user’s role.
  • Deactivate any user, including managers.
Last-admin protection The system blocks you from deactivating or demoting the last System Admin. There must always be at least one active System Admin.

11. Audit Log

Open Audit Log from the sidebar. Every meaningful state change in the system appears here as one row. Each row records:

  • Actor — the signed-in user (or “System” for automated jobs).
  • Action — create, update, delete, reset_pin, toggle_status.
  • Entity — user / client / unit / booking / system.
  • Entity name — e.g. “[A1B2C3D4] Booking for John Doe”.
  • Changes — old → new value for each affected field.
  • IP address — where the action came from.
  • Timestamp.

Filters: by date range, actor, entity type, action. Export to CSV via the button top-right.

Use it for Investigating a dispute (“who edited that booking?”), confirming a sweep ran, reviewing a leaving employee’s actions, or producing a POPIA disclosure log.

12. Security dashboard

Open Security from the sidebar. The page has four tabs:

Failed sign-in attempts
Every wrong PIN attempt, with email, IP and timestamp. Spot brute-force probes by looking for many attempts from one IP across many emails.
Active sessions
Everyone currently signed in. You can revoke a session: the user is kicked out within 60 seconds. Useful when a device is lost or a contractor leaves.
Suspicious activity
Automated flags for things like a session signing in from two different countries in a short window, or a sudden burst of activity from a normally-quiet account.
Sign-in history
Every successful sign-in, with email, IP, user-agent, and timestamp. Filterable.

13. Payments: reconciliation & manual confirm

Most payments resolve automatically through PayFast’s ITN webhook. When that fails (e.g. patient closed the tab too quickly), you have two tools:

Manual confirm

From a single booking’s page, click Confirm payment manually. You will be asked for your PIN as a safeguard. Use this only after you have independently verified the payment landed (PayFast dashboard, EFT proof, etc.).

Reconcile sweep

The reconcile sweep queries PayFast’s Transaction History API and updates any “In Progress” or “Abandoned” bookings from the last 2 hours that PayFast says are actually paid. This runs automatically on the server’s 15-minute schedule (see Operations below), so you normally don’t trigger it yourself. Idempotent — safe to run repeatedly.

Automation A 15-minute cron on the server runs reconciliation automatically (live since 2026-05-27), so in normal operation you rarely need to trigger it by hand. The manual route stays available for one-off checks.

14. Privacy & POPIA erasure

When a patient lodges a POPIA right-to-erasure request, you process it via the privacy API: POST /api/admin/privacy/erase with the patient’s ID number in the body. The system:

  • Anonymises every booking the patient ever had (names, ID, contact, address → null or generic placeholder).
  • Keeps booking IDs and unit_id intact for billing reconciliation.
  • Writes an audit log entry under your name as the actor.
  • Returns a count of how many records were anonymised.
Retention & the unit ID Stakeholders have accepted the residual re-identification risk of keeping unit_id after erasure (it is not personally identifying on its own); the privacy notice documents this. Separately, abandoned (never-paid) bookings are automatically anonymised after 30 days by the retention sweep below — you don’t need to action those manually.

15. Operations & cleanup

A 15-minute cron on the server (live since 2026-05-27) runs the system’s housekeeping automatically — you don’t normally touch any of this by hand. Each tick runs four jobs in sequence:

  • Payment reconcile (/api/payfast/reconcile) — catches any missed PayFast payments.
  • Cleanup sweep (/api/admin/cleanup/sweep) — deletes used/7-day-expired PIN-reset tokens, failed sign-in attempts > 90 days, and resolved incidents > 90 days.
  • Retention sweep (/api/admin/privacy/retention-sweep) — anonymises PII on abandoned (never-paid) bookings older than 30 days (POPIA §14). Completed/paid bookings are left alone — they fall under multi-year medical-records retention.
  • Incident sweep (/api/admin/incidents) — auto-resolves incidents whose symptom has stopped.

Each cleanup-sweep run writes an audit-log entry from actor “Cron”. The cleanup-sweep route also accepts {"dryRun": true} if you ever want to preview what would be deleted without deleting it. Operational setup details (the cron file, the secret, how to verify ticks) live in OPERATIONS.md.

16. PIN verification — who can authorise what

The system has a two-tier PIN verification model. Both tiers use the same PIN modal but accept different roles depending on the action’s trust level:

ActionWho can enter a PIN to authorise
Nurse verification at the start of the booking flow Any Active User, Unit Manager, or System Admin assigned to the unit
Start Consult handoff to CareFirst Any Active User, Unit Manager, or System Admin assigned to the unit
Mark Payment as Confirmed (supervisor override) Unit Manager or System Admin only
User Management edits (edit / delete / reset PIN / change role) Unit Manager or System Admin only
Client Management deletion Unit Manager or System Admin only
Unit Management deletion Unit Manager or System Admin only

As a System Admin your PIN authorises any of the above, anywhere, regardless of unit assignment. Every PIN entry — whether it succeeded or failed — is recorded in the Audit Log against the authoriser’s name.

Why two tiers The lower tier (booking flow) keeps a clinic operating with just one User per unit — no need to pull a manager off another task to sign off every patient. The higher tier (financial overrides + management actions) preserves the two-person sign-off where it genuinely matters for accountability.

17. Common admin problems

ProblemWhat to do
A user reports a payment showing “In Progress” but they paid Open the booking and click Reconcile this booking (or POST /api/payfast/reconcile with that bookingId). The system will query PayFast and update if found.
I see a flood of failed sign-ins from one IP Security → Failed attempts. If it’s clearly hostile, add the IP to the trusted-IPs deny list or block at the firewall. After 5 failed PINs per email the system auto-locks for 15 minutes.
An ex-employee is still signed in somewhere Security → Active sessions → Revoke. Then User Management → toggle their status to Inactive so they can’t sign in again.
Recurring CareFirst hand-off failures Escalate to the CareFirst / development team. The booking stays at “Payment Complete” until the upstream is fixed and Start Consult is retried — no payment is lost.
Need to re-brand a client Client Management → click client → Branding tab → upload new logo / favicon / colour. Changes apply on the next page load for everyone.

18. Things to be careful with

  • Deleting a client cascades through units, users, and bookings. Almost always you should deactivate, not delete.
  • Deleting a unit permanently removes its bookings. Again — prefer deactivation.
  • Changing a client’s payment mode mid-day will confuse users in mid-booking. Schedule outside business hours.
  • Demoting yourself is allowed only if another active System Admin exists. Don’t leave the system without an admin.
  • Manually rotating production secrets (PayFast keys, Supabase service-role key, SMTP password) requires a redeploy. Coordinate with whoever runs the VPS.
  • Running cleanup sweep without dryRun first on a brand-new database can surprise you. Always dry-run once before the first live run.
