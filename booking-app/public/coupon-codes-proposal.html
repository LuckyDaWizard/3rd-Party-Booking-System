Coupon Codes & Discounts
on the Booking Flow

The proposal is to add a discount mechanism to the booking flow. The right shape of that mechanism depends almost entirely on who the discount is for and how it gets decided. This document lays out the four realistic use cases, what each one actually requires to build, and the gating question we need to answer before any work starts. None of this is built yet.

What we know today

  • The consultation fee is currently a single fixed amount per booking, paid via PayFast (or self-collect / monthly-invoice for clients on those modes).
  • There is no discount, voucher, or comp facility anywhere in the flow.
  • The booking row already stores the amount paid — any discount would need to be reflected there and in the audit log so we can reconcile correctly.

The four use cases — pick the one(s) that match the intent

Option A

Marketing & promo codes Real coupons

Anyone with the code can use it (e.g. WINTER20, LAUNCH50). Patient types the code on the payment screen; it’s validated and applied before PayFast is called.

What it needs:

  • A coupons table (code, type=%/fixed, value, valid_from, valid_to, max_uses, used_count, scope=all/client/unit, status)
  • An admin UI to create / list / disable codes (probably a new section under the admin area)
  • An “Apply coupon” input on the payment step that validates server-side
  • Server-side calculation of the discounted amount + record of the coupon on the booking + audit-log entry
  • Refund handling needs to be aware of the discounted amount (PayFast refund uses the actual paid amount, not the original fee)

Effort: ~2–3 days.

Option B

Staff comps & goodwill discounts Manager override

A nurse or manager wants to give a specific patient a discount — usually for goodwill (long-time patient, complaint resolution, friends-and-family). No public code; the operator clicks an “Apply discount” button on the payment step and a manager PIN authorises it.

What it needs:

  • An “Apply discount” button on the payment screen, behind a manager-PIN modal (same gate we use for “Mark Payment as Confirmed”)
  • An input for the discount amount (or a small list of preset percentages: 10% / 25% / 50% / 100% free)
  • Reason field (free text or dropdown) for audit
  • Audit-log entry with the authorising manager, amount, reason

Effort: ~1 day.

Option C — if this is corporate pricing

Per-client / per-unit pricing override Smallest build

Some clients always pay a different rate — e.g. corporate health programme employees always pay R250 instead of the standard fee. No code, no override; the price is just different for every booking under that client.

What it needs:

  • One column on clients: consultation_fee_override (nullable; if NULL, use the system default)
  • Booking creation reads it from the client row instead of the system default
  • That’s it — no UI complexity, no coupon table, no audit-log changes

Effort: ~half a day.

Option D

All of the above

Marketing codes (A) and goodwill discounts (B) are genuinely different features (public vs. internal), and per-client pricing (C) is a different mechanism again. If we need all three, they’re built independently.

Effort: ~3–4 days end-to-end.

If unsure, start with C Per-client pricing override covers the most common real-world reason teams ask for “discount codes” (corporate / wellness pricing) without the heavy machinery of a proper coupon system. We can layer A or B on later if a different need actually emerges.

Questions to nail down before any build

  1. Who is the discount intended for — patients (public codes), specific patients (staff comp), or specific clients (corporate pricing)?
    Determines which option(s) above we’re actually building.
  2. If A (promo codes): single-use or reusable? Time-bound? Limited per client / unit, or system-wide?
    These choices shape the schema and the validation rules.
  3. If B (goodwill comp): is full-comp (R0) allowed, or capped at a percentage?
    Full-comp bookings still need a paid path through PayFast for the record — or skip PayFast entirely — which we need to decide before wiring it.
  4. Discount type: percentage, fixed amount, or both?
    Both is fine but increases the admin UI surface a little.
  5. PayFast refunds: should refunds be on the discounted amount (what the patient paid) or the original fee?
    Refunds should default to the actually-paid amount, but worth confirming.
  6. POPIA & T&Cs: do we need to disclose discount usage anywhere?
    Probably not, but worth flagging.
What we’d need to move forward Confirmation of which option(s) to build (A / B / C / D), plus a quick answer to Q2 or Q3 if the chosen option is A or B. Once that lands the build itself is straightforward.
