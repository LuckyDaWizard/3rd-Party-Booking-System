High-priority findings

H1 — Money-path audit writes are fire-and-forget corroborated ×2

High

api/payfast/notify/route.ts:255 · reconcile/route.ts:318 · start-consultation/route.ts:386

Issue: writeAuditLog() isn't awaited on these payment-confirmation / successful-handoff paths. On standalone Next.js the runtime can tear the request down after the response flushes, dropping the audit insert.

Impact: A confirmed payment or successful handoff may have no audit row — exactly the events most needed for POPIA / financial accountability.

Fix: await all three writes (they're already inside awaited handlers).

H2 — ITN accepts a COMPLETE payment with no amount_gross corroborated ×2

High

api/payfast/notify/route.ts:191-197 · mirror in reconcile/route.ts:264-271

Issue: Amount validation is gated on if (amountGross && …), so a COMPLETE ITN that omits the amount skips the tamper check. In sandbox the signature step accepts without passphrase, so steps 1+3 are weaker than they look.

Impact: Defence-in-depth hole on payment integrity. Must close before real money flows.

Fix: Reject when payment_status === "COMPLETE" and amount is missing. Same for reconcile.

H3 — A unit-manager can assign a user to units they don't own

High

api/admin/users/[id]/route.ts:118-153

Issue: The guard checks the target's current units, but the unitIds in the PATCH body is applied wholesale with no check the caller belongs to each requested unit.

Impact: A unit-manager can move staff into a foreign unit, expanding their reach beyond their authority.

Fix: For non-admins, intersect body.unitIds with caller.unitIds and reject anything outside it.

H4 — Browser-side discard can discard terminal (paid) bookings

High

src/lib/booking-store.tsx:605-612 (enabled by migrations/011:78,104)

Issue: discardBooking does a direct status update with no status guard (unlike abandonBooking). The DB trigger only blocks transitions into paid states, not out of them.

Impact: A paid / handed-off booking can be silently Discarded — distorting reports and dropping it into the 30-day anonymisation set.

Fix: Route discard through a server endpoint via transitionStatus; add a .eq("status","In Progress") guard + a DB OLD.status terminal guard.

H5 — Identity lock is bypassable on the Verify step

High

create-booking/patient-details/page.tsx:1374-1432

Issue: Step 1 locks identity fields to protect an established CareFirst patient, but the Step-4 "Verify your Details" screen re-renders the same fields fully editable, and edits are auto-saved.

Impact: Defeats the lock and reproduces the exact "already registered to a different account" handoff failure it exists to prevent.

Fix: Pass readOnly={identityLocked} to the verify-step identity inputs.

H6 — deleteUser swallows failures → false "deleted" banner

High

src/lib/user-store.tsx:271-276 · user-management/manage/page.tsx:462-468

Issue: deleteUser logs and returns on a non-OK response instead of throwing (unlike deleteClient, which was fixed to throw). The page then shows a success banner regardless.

Impact: On a 500 / RLS denial the user stays in the DB but the operator is told it was deleted — a false success on a destructive action.

Fix: Make deleteUser throw on !res.ok, mirroring deleteClient.

H7 — No monitoring / alerting (app down, cron stopped, payments failing)

High

Operations-wide

Issue: /api/health drives Docker auto-restart, but nothing pages a human. Incidents and logs are pull-only.

Impact: A quiet overnight failure can strand a paid booking with nobody aware until a patient complains.

Fix: External uptime monitor on /api/health + a dead-man's-switch pinged at the end of booking-cron.sh. ~30 min, no code change.

H8 — DB backups documented but not verified as deployed

High

OPERATIONS.md — "Verify Supabase Backups"

Issue: The runbook documents a pg_dump cron, but there's no evidence it's installed on the VPS.

Impact: For a live pilot holding patient PII + payment records, an unverified backup is the highest-consequence gap after alerting.

Fix: Confirm the live Supabase tier; if Free, install the pg_dump cron now and run a restore test once.

What's solid

  • Authorization is uniform and enumeration-resistant — every route uses the correct guard; account-state issues collapse to one generic 403 with the real reason logged server-side only.
  • Service-role routes faithfully replicate RLS in code — they re-check unit membership and server-authority flags before every write rather than trusting the client.
  • PII discipline in logs is excellent — patient PII is redacted from stderr; full context goes only to the RLS-protected audit_log.
  • PIN flows are textbook — salted SHA-256 tokens, 15-min single-use, timing-safe compare, all-session revocation on reset, shared DB throttle.
  • Payment confirmation is robust — constant-time signature compare, fail-closed IP checks, 5s server-confirm abort, correct 200/4xx/5xx retry policy, and triple-layer idempotency so duplicate ITNs and ITN-vs-reconcile races settle cleanly in Postgres.
  • Clean RLS cutover; the trigger rule is honoured without exception — every SECURITY DEFINER function pins search_path; partial-index discipline is consistent.
  • Strong deploy / rollback story — image-tag-per-deploy, DB-verifying healthcheck, log rotation, compose drift resolved, env via SSH .env.
