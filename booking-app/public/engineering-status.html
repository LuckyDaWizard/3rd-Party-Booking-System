B1 — Per-client CareFirst SSO routing Shipped

2026-06-11 · dbd8419 · migration 042 (applied + deployed) · FCS + ISOS live-verified

The last real architecture gap closes. Until now every booking handed off to a single CareFirst account (one CAREFIRST_CLIENT_CODE env var) regardless of which client owned the unit. Now each booking routes to its owning client's CareFirst account, resolved via booking.unit_id → unit.client_id → clients, falling back to the global env default for un-mapped clients (so rollout is incremental and safe). Secret scheme = B-ENV: each client's API key lives in the SSH .env keyed by its CareFirst code (CAREFIRST_API_KEY__<CODE>); only the NON-secret routing fields (carefirst_client_code / _plan_code / _api_domain) are stored on clients (migration 042, nullable, CHECK ^[A-Z0-9]+$). Chosen over a DB-stored key because the client list is read in the browser via .select("*") — a key column would leak — and all other secrets already live in .env. FAIL CLOSED (patient-routing integrity): a mapped client whose key is missing throws rather than silently falling back to the default key — never register a patient under the wrong CareFirst account. The resolver runs before the handoff lock so a misconfig can't strand a booking in handoff_status="in_progress", and it re-validates the code charset before deriving the env-var name (defense-in-depth beyond the DB CHECK, per Code Review). UI: a CareFirst Integration section on Manage Client → Settings sets the codes, with an info banner that the API key is set server-side via SSH (never in the UI). carefirst_client_code (CareFirst's code, e.g. 4B0B68ADC6) is distinct from client_code (our PayFast prefix). Tests: resolver matrix incl. the fail-closed throw, admin validation, end-to-end handoff (mapped / un-mapped / fail-closed) against the mock. Code Review: ship (one High — the charset re-check — fixed). Migration 042 applied + deployed; FCS and International SOS mapped and live-verified 2026-06-11 (Local Choice + Zoie remain on the env-default account). Follow-up: the compose service gained env_file: .env so each new per-client key is just a .env edit; the free-text "API Domain" field was removed from Manage Client after it invited API-keys being pasted into it.

Per-client client_code + readable PayFast m_payment_id (B7 Q1) Shipped

2026-06-11 · 6e99eed · migration 041 (applied + deployed)

Resolves PayFast cutover question Q1 (m_payment_id strategy). Until now the PayFast reference was the bare booking UUID — opaque in the merchant dashboard / settlement reports, with no way to tell which client a payout line belongs to (and the pilot uses one shared merchant account for all clients). New dedicated clients.client_code (3–5 uppercase alnum, unique, nullable) becomes the reference prefix: m_payment_id = "<CODE>-<booking-uuid>", falling back to the bare UUID when a client has no code yet (never hard-errors a payment). The code is set in the Add Client wizard (auto-suggested from the client name, editable, live uniqueness check) and editable on Manage Client — existing clients are filled in over time, no name-based backfill. The load-bearing safety mechanism: a shared stripBookingId() recovers the booking id from BOTH the new prefixed refs and legacy bare-UUID refs, applied at the two confirmation paths (ITN notify + Transaction History reconcile), so in-flight pre-deploy transactions keep confirming; ITN signature validation still runs on the raw posted fields before the strip. Initiate resolves the code via the existing booking→unit→client query (no extra round-trip on the 1-vCPU box). Admin routes validate + uppercase the code and map a 23505 collision to 409. New shared src/lib/client-code.ts. client_code doubles as the future B1 multi-client routing key. Tests: seed client coded TST so initiate emits a prefixed ref; all PayFast/coupon m_payment_id assertions updated; new legacy bare-UUID back-compat ITN test; new pure-lib + admin-validation specs. Code Review: ship, no blockers. Migration 041 applied to Supabase + deployed to the VPS 2026-06-11 (reference docs updated: glossary + payfast-payment-didnt-reflect runbook). Live against sandbox ahead of the credential cutover.

D19 — Duplicate-booking safeguard (double-click defence) Shipped

2026-06-09

Production incident closed: two identical bookings for the same patient at the same minute (Lucky Mokoena, 2026/06/01 14:14, Elizabeth Street unit) traced to a double-click on the create-booking page's Next button. React's batched state updates meant the second click landed BEFORE the disabled={submitting} prop applied; two concurrent createBooking() calls both ran the Supabase insert. Two-layer defence-in-depth fix: (1) Page-level synchronous guard — useRef<boolean> read+written at the very top of the onClick handler, BEFORE any await. If a create is already in-flight, the second caller returns the SAME Promise and resolves with the SAME id. Audit log fires exactly once. Latent bug also fixed: original onClick had inline setSubmitting(false) at each early-return but NONE in the error path — a thrown createBooking() would leave the button stuck disabled forever. New try/finally wrap resets state on every exit (success, return, throw). Code Review caught a related fix (clear verificationCode on PIN failure so a retry can't silently resubmit the same wrong PIN) and flagged D20 follow-up (move to server-authority insert endpoint with rate limit + idempotency key).
