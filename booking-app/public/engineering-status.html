Solution sketch: add carefirst_client_code + carefirst_plan_code to the clients table, UI on the Manage Client · Settings tab, resolve per-booking via unit.client_id, env-var fallback for the transition period. Same underlying problem as the original B4. Half-day
B6 PayFast reconcile 401 in sandbox — expected behaviour, not a bug. Diagnosed 2026-06-05. The reconcile incident payfast:reconcile:query-failed turns out to be a PayFast sandbox quirk — the Transaction History (Query) API returns 401 "Merchant authorization failed" against sandbox credentials even when the same credentials work for payment process + ITN. We’re on sandbox during pilot (PAYFAST_TEST_MODE=true with the public sandbox merchant ID/key/passphrase), so this 401 is harmless: real payment confirmation still flows via the ITN webhook (which works fine on sandbox), and reconcile is only the safety net for ITN drops. Will start working once we promote to live PayFast credentials. See B7. No action
B7 Promote PayFast integration to production credentials. Currently on sandbox during pilot — every "payment" is a sandbox test transaction; no real money flows. When ready to go live:

  3. Update the PayFast merchant dashboard with the production return / notify / cancel URLs — all under https://bookings.carefirst.co.za/api/payfast/....
  4. SSH to the VPS and rebuild: cd /opt/3rd-Party-Booking-System/booking-app && docker compose up -d --build.
  5. Smoke test with a small real-money payment end-to-end; verify ITN arrives + booking flips to Payment Complete + audit log records it correctly.
