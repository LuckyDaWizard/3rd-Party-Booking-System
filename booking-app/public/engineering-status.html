D19 — Duplicate-booking safeguard (double-click defence) Shipped

2026-06-09

Production incident closed: two identical bookings for the same patient at the same minute (Lucky Mokoena, 2026/06/01 14:14, Elizabeth Street unit) traced to a double-click on the create-booking page's Next button. React's batched state updates meant the second click landed BEFORE the disabled={submitting} prop applied; two concurrent createBooking() calls both ran the Supabase insert. Two-layer defence-in-depth fix: (1) Page-level synchronous guard — useRef<boolean> read+written at the very top of the onClick handler, BEFORE any await. Synchronous = blocks the second invocation even within the same render frame as the first. (2) Store-level Promise dedup — useRef<Promise<string> | null> in useBookingStore.createBooking(). If a create is already in-flight, the second caller returns the SAME Promise and resolves with the SAME id. Audit log fires exactly once. Latent bug also fixed: original onClick had inline setSubmitting(false) at each early-return but NONE in the error path — a thrown createBooking() would leave the button stuck disabled forever. New try/finally wrap resets state on every exit (success, return, throw). Code Review caught a related fix (clear verificationCode on PIN failure so a retry can't silently resubmit the same wrong PIN) and flagged D20 follow-up (move to server-authority insert endpoint with rate limit + idempotency key). Pattern captured in feedback_double_click_guard memory for reuse on any future interactive submit handler. 45 / 45 Playwright tests still passing in 3.1m; tsc + lint clean.
