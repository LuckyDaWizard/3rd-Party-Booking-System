In-Flow Consultation Scheduling
VHC Integration — Technical Notes & Questions

Status update — 2026-06-08: Per management decision, in-flow consultation scheduling is deferred. Bookings remain walk-in / same-day only; consultations are not scheduled in advance through the booking system. The existing Start Consult SSO handoff continues to be the integration point with the VHC consultation platform.

The proposal below is preserved as the original scoping document — useful reference if the decision is ever revisited or if a related conversation surfaces with the CareFirst Patient team. For the current booking flow (no scheduling step), see the Booking Flow walkthrough.

Today the booking system handles patient intake and payment, then hands the patient over to the VHC consultation platform via the existing Start Consult SSO handoff. The patient does not currently leave the booking flow knowing the exact date and time of their consultation. This document scopes adding a scheduling step inside the flow so the patient ends with a confirmed slot. It covers where the step fits, the two integration approaches, the API contract we’d need, and the technical questions VHC must answer before we can finalise a design.

The goal

A patient (or the nurse capturing the booking) should be able to see available consultation times, pick one, and have it confirmed as part of the same booking session — before they walk out the door. The confirmed time should appear on the booking confirmation screen and, ideally, in any confirmation email.

Where the step fits in the current flow

The booking flow today runs: patient search → patient details → patient metrics → payment → success → Start Consult (SSO handoff to VHC). The new scheduling step slots in at one of two points depending on whether VHC can hold a slot:

  • Reserve-then-pay: scheduling goes between patient metrics and payment — pick & hold a slot, pay, then the hold is confirmed on success.
  • Pay-then-pick: scheduling goes on the success screen, after payment clears, before Start Consult.

Either way the chosen slot ID is stored on the booking row (a new consultation_slot_id / consultation_at column pair) and passed in the existing Start Consult handoff payload so VHC associates the booked slot with the patient record it receives.

Why this isn’t a simple drop-in

There are two genuinely tricky parts. Both need answers from the VHC side before we can commit to a design.

Challenge 1 — Holding the slot during payment

If the patient picks a 10:30 slot and then spends 90 seconds paying via PayFast, someone else at another clinic could grab that same 10:30 slot in the meantime. There are two ways to handle this:

  • Reserve-then-pay: the slot is held for a few minutes while the patient pays, then confirmed. This needs VHC to support a temporary “hold” on a slot.
  • Pay-then-pick: the patient pays first, then chooses from whatever is still available. Simpler and safer technically, but the patient only learns their time after paying — which slightly weakens the “confirmed time up front” experience.

Challenge 2 — What VHC actually gives us to work with

This is the big fork in the road, and it decides almost everything about cost and quality:

  • If VHC has scheduling APIs (a way for our system to ask “what slots are free?” and “book this one”), we can build the calendar natively inside our booking flow — using our own branding, colours, and mobile-friendly design. The patient never feels like they left our system.
  • If VHC only has a hosted booking page (a web page on their side), our only option is to embed that page in a frame inside our flow. It works, but it will look and feel like a different product bolted on — different fonts, spacing, and colours — and is awkward on mobile.

API contract we’d need (Option B)

For a native build, the minimum surface from VHC is three calls. Shapes below are illustrative — we’ll conform to whatever VHC already has; this just shows the capabilities required.

CapabilityWhat we send / get back
1. Query availability
GET		 Send: client/plan code, date range. Get back: a list of free slots, each with a stable slot_id and an ISO timestamp (and ideally practitioner / duration).
2. Hold a slot optional
POST		 Send: slot_id. Get back: a hold token + expiry (e.g. 5 min). Only needed for the reserve-then-pay flow. If absent, we fall back to pay-then-pick.
3. Confirm / book
POST		 Send: slot_id (or hold token) + the patient identifier we already use in the SSO handoff. Get back: a confirmed booking reference + final timestamp.

Auth would follow the same pattern as the current handoff — the existing CAREFIRST_API_KEY / client-code credentials, server-to-server, never exposed to the browser. The calendar UI calls our own API routes, which proxy to VHC, so the key stays server-side.

Option A technical note (iframe)

If we embed VHC’s hosted page, two things have to be true on VHC’s side:

  • Framing must be permitted from our domain — VHC’s X-Frame-Options / Content-Security-Policy: frame-ancestors headers must allow our booking origin, otherwise the browser blocks the embed outright.
  • A way to know the slot was picked — the embedded page needs to emit a postMessage event (or redirect to a callback URL with the slot details) so our flow can capture the chosen slot and store it on the booking. Without this we’d have no reliable signal that scheduling completed.

Questions for the VHC technical contact

These are the technical facts we need before we can choose an approach or give a firm estimate.

  1. Does VHC expose a scheduling / availability API? VHC
    i.e. can our system programmatically ask “what consultation slots are free for this client/plan on this date?” and receive a machine-readable answer.
  2. Is there a way to book / confirm a specific slot via API? VHC
    And does that booking tie back to the same patient record we hand over at “Start Consult”?
  3. Can a slot be temporarily held / reserved (e.g. for 5 minutes) while the patient pays? VHC
    This decides whether we can do “reserve-then-pay” or have to fall back to “pay-then-pick”.
  4. If there is no API — is there a hosted booking page we could embed? VHC
    And critically, can VHC allow that page to be embedded from our booking domain (a security-header / X-Frame-Options change on their side)?
  5. How is availability scoped — per practitioner, per clinic, per client? VHC
    So the patient is only ever shown slots that are genuinely valid for their booking.
  6. Does VHC send its own confirmation (email/SMS) once a slot is booked? VHC Answered
    Determines whether we need to send the confirmed time ourselves or can rely on VHC to notify the patient.
    Yes — VHC sends email/SMS confirmation to the patient when a slot is booked. Implication: we don’t need to build our own confirmed-time notification; our booking confirmation screen just needs to display the time, and VHC handles the patient reminder. One less thing to build.
Next technical step Get answers to Q1–Q6 from the VHC side. Q1–Q3 (does an availability/booking API exist, and can a slot be held) are the load-bearing ones — they decide between Option A and Option B and whether we can do reserve-then-pay. Once those land we’ll finalise the integration design and give a firm estimate (indicatively 2–3 days native, ~half that for an embedded page).
