01 - Context Why this audit was done

The CareFirst handoff outage that triggered this review has since been resolved on the upstream side. With the system back to a steady state, this audit takes a step back and looks at the whole platform - what is solid, what is brittle, what is a risk to the business, and what is just maintenance dust.

The audit was conducted by three parallel reviewers each focused on a different lens (security, dead code, performance) over the entire codebase including all source files, database migrations, server logs, and configuration. Approximately 90 distinct findings were captured. This document groups them by business impact and recommends an order of attack.

02 - Summary Executive summary

Overall health Good. The system is well-architected for its current scale. The work done over the past few months on security hardening, audit logging, and incident detection has visibly raised the floor.
Top blocker for production The application is still running on plain HTTP rather than HTTPS. Every login, every patient record, every payment confirmation travels across the network unencrypted. This is the single biggest risk and the one item that should be fixed before anything else.
POPIA exposure Patient names and ID details are written into the server's diagnostic logs in a few places. Logs are not patient-medical-record-grade storage and this puts the company at risk of a compliance finding. Easy to fix.
Scalability ceiling Today: comfortable up to ~1,000 bookings per unit. Beyond that, a handful of database queries and the bookings list page start to slow visibly. Nothing is broken today - these are problems worth solving before they bite.
Dead code Surprisingly little. The team has been disciplined about removing features cleanly (scheduling, old PIN columns). Three minor housekeeping items only.
Operational gaps The system has no automatic background schedule for payment reconciliation, incident cleanup, or old-record purging. Today these run only when a user happens to trigger them. Fine at low volume; needs a small scheduler before scale.

03 - Positives What's working well

Worth highlighting before the issues list so the picture isn't lopsided.

Comprehensive audit trail

Consistent role separation

No dead dependencies

Strict type safety

Lean Docker setup
Multi-stage build, healthchecks in place, correct restart policy.

04 - Critical Fix before next external release

Three items here. They are all known to the team; this audit confirms they remain the top priorities.

1. Site runs over plain HTTP, not HTTPS

What it is
Visitors connect to http://187.127.135.11:3000 rather than https://. There is no TLS certificate, no encryption.
Why it matters
Every patient record, every login PIN, every payment confirmation, every session cookie travels across the network in clear text. Anyone in a position to observe the network (public Wi-Fi at a clinic, an upstream ISP, a malicious actor on the same network) can read or tamper with it. POPIA explicitly requires encryption of patient data in transit. This is non-negotiable for go-live.
What to do
Provision a real domain name (e.g. booking.firstcare.solutions), obtain a TLS certificate (Let's Encrypt is free and automated), point the domain at the VPS, and update the environment variable that builds outbound links. The application code is already prepared - the cookie security flag and CSRF cookie security flag both flip on automatically the moment HTTPS is detected.
Effort
Half a day, mostly waiting for DNS propagation.
Why half a day

Tech work is trivial (~30 minutes): provision domain, run certbot for Let's Encrypt, update Traefik. The day is consumed by:

  • DNS propagation wait (15 minutes - 2 hours depending on the TTL).
  • Updating NEXT_PUBLIC_APP_URL and re-deploying.
  • Updating Supabase allowed redirect URIs.
  • Testing the full PayFast callback round-trip - their notify URL must be HTTPS.
  • Verifying email links (sign-in magic link, PIN reset) still resolve correctly.
  • Smoke-testing sign-in across 2-3 devices.

2. Patient details appear in server diagnostic logs

What it is
When a Start Consult handoff fails (e.g. the CareFirst 502 last week), the system writes the patient's first name, ID type, nationality, gender, country, and province into the server log so a technician can diagnose the problem. PayFast payment notifications also log the booking's unique ID.
Why it matters
Server logs are streamed to operational tooling (Docker logs, Hostinger monitoring, any future log aggregator), retained for varying periods, and accessible to anyone with VPS console access. They are not designed for patient-medical-record handling. POPIA's data-minimisation principle says we should only retain what we need, in the right place, for the right people.
What to do
Replace the patient-details snapshot with a hash or the short booking reference. Move detailed failure diagnostics into the existing audit_log table (which IS designed for this) or into a dedicated secure-error endpoint that requires admin sign-in to read.
Effort
2-3 hours.
Why 2-3 hours

The code change itself is ~15 minutes per site, and there are only 2 sites. The rest of the time covers:

  • Triggering a fake handoff failure locally to confirm what gets redacted (~20 min).
  • Deciding the redacted format - hash vs short reference ID vs booking ID only (~15 min).
  • Confirming the equivalent detail is still recoverable from the audit_log table (where patient detail belongs) (~30 min).
  • Spot-checking past log retention to confirm the leak window closes once redacted (~30 min).
  • Buffer for deploy + sanity check in production logs (~30 min).

3. Confirm the secret-config file isn't in git history

What it is
The reviewer flagged that the file holding production API keys, SMTP credentials, and payment-gateway secrets (.env.local) may have been committed to the repository at some point. If so, anyone with a copy of the repository (including from any historical fork or backup) can read those secrets even if the file is now removed.
Why it matters
Committed secrets cannot be "uncommitted" in git history without rewriting the history. If they were ever pushed, they have to be assumed leaked and rotated.
What to do
Check the git log for the file. If it ever appears: rotate every key it contains - Supabase service role key, PayFast credentials, CareFirst API key, SMTP password - remove the file from git history with the BFG cleaner, and confirm .env.local is in .gitignore. If it was never committed, no action needed.
Effort
15 minutes to check; 2-3 hours of rotation work only if the file was actually committed.
Why this range

15 minutes if the file is clean - one git log --all --full-history -- .env.local command tells you definitively.

2-3 hours worst case if the file is dirty:

  • Rotate 4 secrets - Supabase service role, PayFast credentials, CareFirst key, SMTP password. Each provider has a 10-15 min rotation flow.
  • Run BFG cleaner or git-filter-repo to scrub the file from git history (~30 min).
  • Force-push to remote and coordinate with anyone holding clones (~30 min).
  • Re-deploy with new secrets (~15 min).
  • Verify sign-in, payment notify, CareFirst handoff and email all still work after the rotation (~45 min).

05 - High Address this quarter

These won't break the system tomorrow, but each is a real liability waiting for the right (wrong) day.

4. Database is missing indexes on the columns we query most

What it is
Several columns that the application filters by on every page - for example, the column linking a booking to its unit - don't have a database index. Today the booking volume is small enough that the database scans the whole table fast enough that nobody notices. As bookings grow, those scans get linearly slower.
Why it matters
The breaking point is roughly 1,000 bookings per unit. Beyond that the booking list page and unit-scoped audit queries get noticeably slow. Adding indexes is a one-line change.
What to do
Add database indexes on the join keys for bookings and user-unit assignments. Ship as migration 030.
Effort
1 hour.
Why 1 hour

The migration SQL is 3 lines (one CREATE INDEX per column). The hour covers the surrounding ceremony:

  • Write migration 030 in the existing migration format (~10 min).
  • Apply locally and run EXPLAIN ANALYZE on representative queries to confirm Postgres picks up the index (~15 min).
  • Apply to staging and smoke-test the booking list + audit query paths (~15 min).
  • Apply to production - this is trivial since indexes are additive and cause no lock contention at this scale (~20 min including the verification window).

5. No automatic background schedule for clean-up tasks

What it is
The system has three jobs that should run on a timer but don't:
  • Payment reconciliation - re-checks PayFast for any payment we might have missed. Today only runs when an admin clicks a button.
  • Incident auto-resolve sweep - clears the red banner on the Reports page when failures stop. Today only runs when someone opens the page.
  • Old-record cleanup - old PIN-reset codes, expired sessions, ancient login attempts. Today nothing deletes them.
Why it matters
Payments can sit in limbo if an admin doesn't happen to look. The incidents table will slowly bloat. Old security artefacts accumulate forever. None of this is catastrophic, but the entire premise of "self-healing" relies on regular sweeps.
What to do
Add a 15-minute external cron job that hits the existing reconcile and incident-sweep endpoints, plus a new clean-up endpoint to be built.
Effort
4 hours including writing the cleanup endpoint.
Why 4 hours

Three discrete pieces of work bundled together:

  • Build the cleanup endpoint POST /api/admin/cleanup/sweep - service-role auth gate, three DELETE queries (expired pin_reset_tokens, old sessions, ancient login_attempts), log row counts for observability (~1.5 hr).
  • Choose and configure the cron runner - cron-job.org, GitHub Actions schedule, or Hostinger-native cron. Three sequential HTTP POSTs with proper auth headers (~1 hr).
  • Verify the schedule fires - watch the first few 15-minute ticks land, confirm reconcile + incidents + cleanup all complete (~1 hr).
  • Add alerting if the cron stops firing - silent failures are the worst kind (~30 min).

6. Booking list re-loads everything after every change

What it is
When you create, edit, discard, or abandon a booking, the app reloads the entire list of bookings for the unit from the database. With 50 bookings this is fine. With 500 it flickers and loses scroll position.
Why it matters
This is the most-used page in the application. As volume grows, the UX degrades visibly - every click feels a bit slower than the last. Also wastes database load.
What to do
Patch the local list in place after a successful update instead of refetching. Standard React pattern; ~30 lines of change in the booking store.
Effort
2-3 hours plus testing.
Why 2-3 hours

The code change is small but verification is the heavy bit:

  • Update 4 mutation points in booking-store.tsx to patch local state instead of triggering a refetch (~1 hr code).
  • Manually test each of the 4 paths - create, edit, discard, abandon - verifying scroll position is preserved and no flicker (~1 hr).
  • Edge case thinking: what if another tab is open? Stale data risk - acceptable for this volume but documented (~30 min).

Complete7. Authentication check is duplicated across every protected endpoint

Resolved 2026-05-21 Added a default-deny check to src/middleware.ts: any /api/admin/* request without a Supabase auth cookie now gets a 401 before reaching the handler. The existing in-handler requireSystemAdminWithCaller() guards stay in place as defence-in-depth. Belt + braces: even if a future admin endpoint forgets the role check, unauthenticated traffic is blocked. Coverage enforced by a new Playwright test (tests/admin-auth-coverage.spec.ts) that walks every route.ts under src/app/api/admin/ and asserts each rejects unauthenticated requests with 401/403/405. All 23 endpoints currently pass.
What it is
Every API endpoint that requires sign-in re-runs the same 20 lines of auth-checking code. There are roughly 30 endpoints. If a new one gets added and the developer forgets to copy-paste the check, that endpoint silently has no security.
Why it matters
This is how security holes appear. It's not a current vulnerability - every existing endpoint has been checked - but the pattern is fragile. The risk grows every time a new endpoint is added.
What to do
Move the auth check into a single piece of middleware that runs for any URL under /api/admin/. Then individual endpoints only have to declare "I need admin" once.
Effort
1 day including testing every endpoint after the change.
Why 1 day

The change itself is small but the coverage matrix is large:

  • Write the middleware that runs on the /api/admin/* matcher with role-based checks (~2 hr).
  • Remove the redundant auth blocks from ~30 endpoints (~1 hr - mechanical).
  • Re-test all 30 endpoints. Each one needs three checks: signed-out returns 401, wrong role returns 403, correct role returns 200. Don't skip - auth was previously per-endpoint so any regression is silent (~5 hr).

The day reflects the testing matrix, not the typing.

Complete8. Booking state transitions are scattered across many files

Resolved 2026-05-21 New src/lib/booking-state-machine.ts defines the canonical model: a TRANSITIONS map of allowed target statuses per source, plus canTransition() / assertTransition() validators and a transitionStatus(client, id, from, to, extras) helper that combines validation with a conditional UPDATE. Five server-side transition sites migrated: PayFast ITN webhook, PayFast reconcile sweep, admin complete-payment, mark-monthly-invoice, mark-self-collect, and Start Consult (Payment Complete → Successful). Each now filters its UPDATE by the expected current status, so concurrent writers (e.g. ITN arriving mid-reconcile) settle cleanly at the DB level — the loser gets a "conflict" result instead of double-writing the row + audit log.

Follow-up (also 2026-05-21): a race-test on Start Consult revealed that while the DB transition was protected, both racing requests still called CareFirst (which is not idempotent on uniqueReference), creating duplicate SSO sessions. Added a handoff lock pattern: a conditional UPDATE acquires handoff_status = "in_progress" before the CareFirst call, returning 409 to the loser. Only one window reaches the external API.
What it is
A booking moves through statuses (In Progress to Payment Complete to Successful or Discarded/Abandoned). The rules for which transitions are allowed live in eight different files. There's no single place that says "from status X you can only go to Y or Z".
Why it matters
Two things can go wrong simultaneously - for example, manual payment confirmation by an admin happening at the same moment as automatic PayFast reconciliation - and the system has no central guard to prevent inconsistent states. We haven't been bitten by this yet, but the longer the system runs the more likely it gets.
What to do
Define the state machine in one shared module. Every transition path calls into that module rather than implementing its own checks.
Effort
1-2 days; this is a refactor, not a bug fix.
Why 1-2 days

The risk profile is the driver, not the keystroke count:

  • Map all transition logic currently scattered across booking-store.tsx and 8 API routes (~2 hr).
  • Define the state machine in one module - states + the allowed transitions between them (~3 hr).
  • Replace inline checks with calls to the shared module (~3 hr).
  • Race-condition testing - this is the big chunk. Admin manual-confirm racing with PayFast auto-reconcile, abandoned-flow racing with continue, payment-failed racing with success. Each race needs reasoning + reproduction (~4-8 hr).

It's marked 1-2 days because the lower bound assumes no surprising race interactions and the upper bound assumes one or two pieces of logic that turn out to depend on timing in subtle ways.

9. POPIA erasure leaves the unit ID behind

What it is
When a patient requests deletion under POPIA, the system anonymises their record but keeps the unit identifier for billing reconciliation. The intent is good, but unit + erasure timestamp + booking history can in theory be used to re-identify the patient.
Why it matters
The compliance promise is "we deleted everything that identifies you". The current implementation can be challenged on that promise.
What to do
Decide as a business: either accept the residual re-identification risk and document it in the privacy notice, or replace the unit reference with a generic billing code that's less granular.
Effort
Half a day plus a stakeholder conversation.
Why half a day

The work is mostly the conversation, not the code:

  • Read the current implementation of /api/admin/privacy/erase to understand exactly what's anonymised and what's retained (~30 min).
  • Stakeholder conversation - legal/compliance officer + business owner. Decide whether to accept the residual re-identification risk OR replace unit_id with a generic billing code (~1-2 hr).
  • If "accept the risk" path: update the privacy notice + document the decision (~1 hr).
  • If "replace" path: update the erase route to swap unit_id for a client-level billing code, test the run end-to-end (~2 hr).

Half a day is the conversation + decision + write-up combined. The code is the small part.

09 - Verify How to verify any finding

This audit was conducted by reading the codebase and the dev server logs directly. To verify any specific claim:

Security claims
Open the referenced file at the referenced line. The behaviour is in the code.
Database claims
In the Supabase SQL editor, run SELECT indexname FROM pg_indexes WHERE tablename = 'bookings' and compare against the columns the app filters on.
Performance claims
Server response times are visible in the dev log. Endpoints over 1 second consistently are worth investigating.
Dead-code claims
Each cited file path is a real location in the repo. Grep for the symbol; if it has zero call sites, the claim stands.

No claim in this document was made without a verified code reference. The reviewers were instructed to flag only what they could verify, not speculate.

10 - Scope What this audit deliberately did not cover

For honesty about scope, these were not in this audit:

  • Penetration testing. This is a code review, not a live attack against the running system. Some vulnerabilities only surface under active probing.
  • Third-party dependency CVE scan. Worth running npm audit separately, ideally as part of CI.
  • Load testing. The "scalability ceiling" numbers above are estimates based on query patterns, not measured under real load.
  • Accessibility audit. WCAG conformance was not assessed.
  • Browser compatibility. Tested in development only; real-world device coverage unknown.

These are all reasonable follow-up audits depending on how the company prioritises risk.
