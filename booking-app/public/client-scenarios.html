Client Scenarios — No-Gateway Booking Walkthrough

Internal Architecture review

How four identified clients map to the system today, and what would make it easier.

Dated 11 June 2026 · Production bookings.carefirst.co.za · Note Updated 2026-06-11 — verified against the current codebase.

00 · Summary The core model

One patient per booking, keyed on ID

The system's unit of work is one patient per booking, identified by their ID/passport number. Whoever is logged in (a nurse, a 3rd-party agent, an admin) always creates the booking on behalf of the patient — so "nurse books" and "3rd party books" are the normal mode, not special cases.

Two architectural facts shape every scenario below: (1) there is no main-member / dependant relationship and no membership/scheme-number field — a dependant is just another standalone patient record keyed on their own ID; (2) per-client CareFirst routing is now LIVE (B1, shipped 2026-06-11) — each booking hands off to its owning client's own CareFirst account, with un-mapped clients falling back to a default account. Mapping a client = set its CareFirst code in Manage Client + add its API key to the server .env.

For "no payment gateway" there are two client-level modes: Monthly Invoice (no payment step; auto-marks Payment Complete; billed month-end) or Self-Collect (unit takes cash/card, operator confirms with PIN), plus optional Skip Patient Metrics and Nurse Verification (PIN sign-off) flags.

All four clients can be onboarded as no-gateway entirely through the Manage Client screen — the Monthly Invoice / Self-Collect / Skip Patient Metrics / Nurse Verification toggles are all self-service, with no database work required.

01 · Modes How the no-gateway modes behave

Two billing modes, mutually exclusive

Monthly Invoice (bill_monthly): the payment step is skipped automatically — the booking auto-marks Payment Complete (payment_type = monthly_invoice) and the operator never sees a payment screen. The client is billed at month-end.

Self-Collect (collect_payment_at_unit): the unit takes the fee; the operator confirms with a "payment collected at unit" action. Note: this confirm is a plain button click unless the client also has Nurse Verification enabled, in which case a PIN gate applies at the verification step (the PIN comes from the Nurse-Verification flag, not from self-collect itself).

The two billing modes are mutually exclusive (enforced in the UI and server-side). Skip Patient Metrics (optional, only meaningful with a no-gateway mode) jumps straight past the clinical-metrics step.

Flow
Member-vs-dependant is invisible to the system — both are identical standalone single-patient bookings, so there's no functional blocker. Each person is booked as a standalone patient by their own ID; the operator (nurse) is just the logged-in user.
Note
The system doesn't model "main member vs dependant" or capture a membership/scheme number, so if NHG needs that linkage recorded it isn't there today (see the roadmap below).

Works — with caveats 3. NWI Schools — nurse books a dependant using the PARENT's contact

Flow
Enter the child's identity + the parent's cell/email.
Blocker (a)
No-ID path for young children (sharpest gap): the ID-type options are only National ID or Passport, and an ID number is required to complete the CareFirst handoff. A young child with neither has no clean path — the operator has to improvise an entry. (There's no "birth certificate / Other / none" option.)
Blocker (b)
Shared parent contact is blocked across siblings: the duplicate-contact check blocks the Next button (it isn't just a warning) when a contact number is already on another patient's booking. So booking a second child on the same parent's number halts the flow until the operator clears it.
Comms
Comms route to the parent (as intended): the child's CareFirst account is registered with whatever contact the booking carries — here the parent's cell/email — so OTPs/communications go to the parent. Nothing labels the contact as a "guardian's."

03 · Takeaway Cross-cutting takeaway

All four work today; NWI has the one hard stop

All four work today with the right Manage-Client configuration (no DB work). Per-client CareFirst routing is now live (B1, shipped 2026-06-11) — FCS and ISOS already route to their own accounts; un-mapped clients (Local Choice, Zoie) use the default until mapped. The one remaining modeling gap is no dependant/membership relationship. The NWI-specific friction (no-ID path + the blocking duplicate-contact check on shared parent numbers) is the one place an operator still hits a hard stop today.
