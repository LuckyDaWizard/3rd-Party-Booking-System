01 - Context Why this audit was done

Closure record — 18 May 2026 baseline
Everything from here down is the original May 2026 audit, worked to closure over the following weeks. Findings are marked Complete or Deferred and are kept as the historical record. For the current state of the system see the 11 June 2026 follow-up above.

The CareFirst handoff outage that triggered this review has since been resolved on the upstream side. With the system back to a steady state, this audit takes a step back and looks at the whole platform - what is solid, what is brittle, what is a risk to the business, and what is just maintenance dust.

The audit was conducted by three parallel reviewers each focused on a different lens (security, dead code, performance) over the entire codebase including all source files, database migrations, server logs, and configuration. Approximately 90 distinct findings were captured. This document groups them by business impact and recommends an order of attack.

02 - Summary Executive summary

Overall health Good. The system is well-architected for its current scale. The work done over the past few months on security hardening, audit logging, and incident detection has visibly raised the floor.
Top blocker for production The application is still running on plain HTTP rather than HTTPS. Every login, every patient record, every payment confirmation travels across the network unencrypted. This is the single biggest risk and the one item that should be fixed before anything else.
POPIA exposure Patient names and ID details are written into the server's diagnostic logs in a few places. Logs are not patient-medical-record-grade storage and this puts the company at risk of a compliance finding. Easy to fix.
Scalability ceiling Today: comfortable up to ~1,000 bookings per unit. Beyond that, a handful of database queries and the bookings list page start to slow visibly. Nothing is broken today - these are problems worth solving before they bite.
Dead code Surprisingly little. The team has been disciplined about removing features cleanly (scheduling, old PIN columns). Three minor housekeeping items only.
Operational gaps The system has no automatic background schedule for payment reconciliation, incident cleanup, or old-record purging. Today these run only when a user happens to trigger them. Fine at low volume; needs a small scheduler before scale.

BHigh-priority findings

Eight items, all open. Each won't break the pilot tomorrow, but each is a real liability concentrated in authorization, audit durability, or operational visibility.

ResolvedH1. Money-path audit writes are fire-and-forgetcorroborated ×2

Where
api/payfast/notify/route.ts:255 · reconcile/route.ts:318 · start-consultation/route.ts:386
Issue
writeAuditLog() isn't awaited on these payment-confirmation / successful-handoff paths. On standalone Next.js the runtime can tear the request down after the response flushes, dropping the audit insert.
Impact
A confirmed payment or successful handoff may have no audit row — exactly the events most needed for POPIA / financial accountability.
Fix
await all three writes (they're already inside awaited handlers).
Resolution
✅ Fixed 2026-06-11 (c529d4e) — all four money-path audit writes awaited (notify + reconcile + start-consultation failed/success). writeAuditLog never throws, so awaiting is safe; each await lands after the state commit. Code Review: ship. Pending deploy.

ResolvedH2. ITN accepts a COMPLETE payment with no amount_grosscorroborated ×2

Where
api/payfast/notify/route.ts:191-197 · mirror in reconcile/route.ts:264-271
Issue
Amount validation is gated on if (amountGross && …), so a COMPLETE ITN that omits the amount skips the tamper check. In sandbox the signature step accepts without passphrase, so steps 1+3 are weaker than they look.
Impact
Defence-in-depth hole on payment integrity. Must close before real money flows.
Fix
Reject when payment_status === "COMPLETE" and amount is missing. Same for reconcile.
Resolution
✅ Fixed 2026-06-11 (9a9aa43) — COMPLETE ITN with missing amount_gross → 400; reconcile refuses a matched COMPLETE txn with no amount — both before the booking flips. Tests added (ITN + reconcile). Code Review: ship. Pending deploy. With H1, both PayFast go-live blockers are cleared.

OpenH3. A unit-manager can assign a user to units they don't own

Where
api/admin/users/[id]/route.ts:118-153
Issue
The guard checks the target's current units, but the unitIds in the PATCH body is applied wholesale with no check the caller belongs to each requested unit.
Impact
A unit-manager can move staff into a foreign unit, expanding their reach beyond their authority.
Fix
For non-admins, intersect body.unitIds with caller.unitIds and reject anything outside it.

OpenH4. Browser-side discard can discard terminal (paid) bookings

Where
src/lib/booking-store.tsx:605-612 (enabled by migrations/011:78,104)
Issue
discardBooking does a direct status update with no status guard (unlike abandonBooking). The DB trigger only blocks transitions into paid states, not out of them.
Impact
A paid / handed-off booking can be silently Discarded — distorting reports and dropping it into the 30-day anonymisation set.
Fix
Route discard through a server endpoint via transitionStatus; add a .eq("status","In Progress") guard + a DB OLD.status terminal guard.

OpenH5. Identity lock is bypassable on the Verify step

Where
create-booking/patient-details/page.tsx:1374-1432
Issue
Step 1 locks identity fields to protect an established CareFirst patient, but the Step-4 "Verify your Details" screen re-renders the same fields fully editable, and edits are auto-saved.
Impact
Defeats the lock and reproduces the exact "already registered to a different account" handoff failure it exists to prevent.
Fix
Pass readOnly={identityLocked} to the verify-step identity inputs.

OpenH6. deleteUser swallows failures → false "deleted" banner

Where
src/lib/user-store.tsx:271-276 · user-management/manage/page.tsx:462-468
Issue
deleteUser logs and returns on a non-OK response instead of throwing (unlike deleteClient, which was fixed to throw). The page then shows a success banner regardless.
Impact
On a 500 / RLS denial the user stays in the DB but the operator is told it was deleted — a false success on a destructive action.
Fix
Make deleteUser throw on !res.ok, mirroring deleteClient.

ResolvedH7. No monitoring / alerting (app down, cron stopped, payments failing)

Where
Operations-wide
Issue
/api/health drives Docker auto-restart, but nothing pages a human. Incidents and logs are pull-only.
Impact
A quiet overnight failure can strand a paid booking with nobody aware until a patient complains.
Fix
External uptime monitor on /api/health + a dead-man's-switch pinged at the end of booking-cron.sh. ~30 min, no code change.

OpenH8. DB backups documented but not verified as deployed

Where
OPERATIONS.md — "Verify Supabase Backups"
Issue
The runbook documents a pg_dump cron, but there's no evidence it's installed on the VPS.
Current state (2026-06-29)
Confirmed still open: Supabase on Free tier (no auto-backups), and the pg_dump cron has not been installed on the VPS. As of the go-live cutover the app holds real production PII and (soon) real payment records with zero backup coverage.

OpenS1. 🚨 Credential exposure — 7 real user PINs committed to git in cleartext

Where
booking-app/scripts/migrate-pins-to-6-digits.mjs:36-45 (HEAD) + git history since commit aa92178.
Surfaced
2026-07-17, targeted credential-exposure audit.
Issue
A one-off PIN-migration script hardcoded 7 real users' 6-digit PINs in the source. Per the project's synthetic-email auth scheme (pin-<pin>@carefirst.local), each PIN is also that user's auth.users password. Anyone with repo read access has valid sign-in credentials for those accounts.
Live blast radius (verified 2026-07-17)
None of the 7 accounts is currently Active: 4 are Disabled (unit_manager role), 3 are not present in public.users. But: disabled users can be re-enabled by a system_admin; the leaked PIN would be instantly valid again. And the 3 "not found" accounts may still exist in auth.users under their synthetic email — if so, sign-in via Supabase auth directly would still work.
Impact
Full account takeover for 4 unit_manager accounts (once re-enabled) and potentially 3 more if they still exist in auth.users. Unit_manager can view / manage bookings within assigned units, create / disable users in their scope, etc. Repo is private so exposure is bounded to authorized team members, but the values are permanent in git history.
Remediation status (2026-07-17)
✓ Script fixed — HEAD now requires PINs via PIN_ASSIGNMENTS env var (JSON array); refuses to run without one. Doc-comment warns future maintainers not to re-introduce hardcoded PINs.
☐ PIN rotation — user action required. Rotate all 7 PINs via scripts/update-user-pin.mjs (or delete/recreate the affected auth users). Rotate even the disabled ones — else re-enabling silently restores exposed access.
☐ Auth.users cleanup — user verification required. In Supabase Dashboard → Authentication → Users, search for the 3 "not found" PINs (pin-487293, pin-740582, pin-529607 at @carefirst.local) and delete any that exist.
☐ Optional git-history purge (git filter-repo) — nice-to-have. Rotation makes the leaked values useless, so this is lower priority. Private-repo audience is bounded.

Where
booking-app/tests/_setup/seed.ts:57, booking-app/OPERATIONS.md:746,757, booking-app/public/engineering-status.html.
Issue
The Playwright seed creates a system_admin fixture user with PIN 900900 (email pin-900900@carefirst.local). Intended for dev Supabase only. If the seed has EVER been run against production Supabase, a system_admin account with a well-known PIN exists in prod.

OpenP1. Capture Vital Signs page uses non-standard units, no ranges, and a broken Urine Dipstick field

Where
src/app/(dashboard)/create-booking/patient-metrics/page.tsx
Surfaced
2026-07-14 during pre-go-live review of the operator flow.
Issue
The metrics form accepts free-text with no clinical validation and has several concrete data-integrity problems:
  • Unit-label typos: Blood Pressure shows hhMG (should be mmHg); Temperature shows celcius (should be °C).
  • US units in an SA product: Glucose is labelled mg/dL — SA / most of the world uses mmol/L.
  • Unrealistic placeholders: Blood Pressure placeholder is 000/00, temperature is 000 — give no clue to the operator about the expected value.
  • Wrong concept — Urine Dipstick: field asks for a single number labelled pH, but a urine dipstick is a multi-parameter panel (glucose, protein, ketones, blood, leukocytes, nitrites, bilirubin, urobilinogen, specific gravity, pH) — not one number, and pH is only one of ten parameters.
  • No range validation anywhere. Operator can save physiologically impossible values (BP 900/500, temperature 999°C, SpO₂ 250%) with no warning — the values persist to the booking row and any downstream export/report.
  • Missing standard vitals: no respiratory rate (a standard adult vital sign), no weight / height / BMI, no pain score.
Impact
Operators can save non-clinical / physiologically impossible values into the patient record; the Urine Dipstick field captures data that isn't structured usefully; the form doesn't match SA clinical practice.
Recommended fix — two options
Option A — Minimal, no schema change (~30 min): Fix unit labels + typos, add realistic placeholders (BP 120/80, temperature 36.5), convert glucose to mmol/L, add soft-warning range validation for each metric, convert Urine Dipstick to a multi-select checkbox panel for the standard 10 parameters (stored as JSON in the existing urine_dipstick text column — no migration needed).

Option B — Comprehensive, needs schema migration (~2 hrs): Everything in A, plus new fields: respiratory_rate, weight_kg, height_cm (with BMI auto-calculated), pain_score. Migration 043 adds the new columns to bookings.
Open questions to answer before implementing
  1. Glucose unit: mmol/L (SA standard) confirmed, or does any client / partner require mg/dL? If mixed, may need a client-level setting.
  2. CareFirst handoff: currently vitals are stored on the booking but NOT sent to CareFirst. If clinicians in CareFirst need these vitals, we should add them to buildSsoPayload (a small extension of the additionalEmail work).
  3. Which option: A (label + range fix now) or B (also add missing vitals + schema change).

04 - Critical Fix before next external release

Three items here. They are all known to the team; this audit confirms they remain the top priorities.

Complete1. Site runs over plain HTTP, not HTTPS

Resolved 2026-06-03 (HTTPS) + 2026-06-08 (HSTS) bookings.carefirst.co.za subdomain provisioned (IT-managed A-record). Traefik configured with the letsencrypt cert resolver via HTTP-01 ACME challenge; certificate auto-renews. Booking-app router labels switched to entrypoints=websecure + tls.certresolver=letsencrypt + Host(\`bookings.carefirst.co.za\`). HTTP→HTTPS redirect configured at the Traefik entrypoint level. As of 2026-06-08, HSTS is also live via Traefik headers middleware (max-age=63072000; includeSubDomains); preload deferred for a one-week soak before submission to the Chromium preload list. Solved the FortiGuard "Unrated IP" office block as a bonus — the subdomain is categorised Business.
What it is
Visitors connect to http://187.127.135.11:3000 rather than https://. There is no TLS certificate, no encryption.
Why it matters
Every patient record, every login PIN, every payment confirmation, every session cookie travels across the network in clear text. Anyone in a position to observe the network (public Wi-Fi at a clinic, an upstream ISP, a malicious actor on the same network) can read or tamper with it. POPIA explicitly requires encryption of patient data in transit. This is non-negotiable for go-live.
What to do
Provision a real domain name (e.g. booking.carefirst.co.za), obtain a TLS certificate (Let's Encrypt is free and automated), point the domain at the VPS, and update the environment variable that builds outbound links. The application code is already prepared - the cookie security flag and CSRF cookie security flag both flip on automatically the moment HTTPS is detected.
Effort
Half a day, mostly waiting for DNS propagation.
Why half a day

Tech work is trivial (~30 minutes): provision domain, run certbot for Let's Encrypt, update Traefik. The day is consumed by:

  • DNS propagation wait (15 minutes - 2 hours depending on the TTL).
  • Updating NEXT_PUBLIC_APP_URL and re-deploying.
  • Updating Supabase allowed redirect URIs.
  • Testing the full PayFast callback round-trip - their notify URL must be HTTPS.
  • Verifying email links (sign-in magic link, PIN reset) still resolve correctly.
  • Smoke-testing sign-in across 2-3 devices.

Complete2. Patient details appear in server diagnostic logs

Resolved 2026-05-22 Removed the payloadSnapshot block from start-consultation/route.ts that dumped the patient's first name, ID type, nationality, gender, country, and province into stderr. Replaced with a short bookingRef hash + CareFirst's own statusCode/error/rawResponse — none of which contain our PII. Full booking context is preserved in the audit_log table where it belongs (service-role RLS protected). Also swept every ${'$'}{bookingId} interpolation in /api/payfast/notify + /api/payfast/reconcile console statements to use bookingRef() instead, so docker-logs streams can't be cross-referenced back to the bookings table.
What it is
When a Start Consult handoff fails (e.g. the CareFirst 502 last week), the system writes the patient's first name, ID type, nationality, gender, country, and province into the server log so a technician can diagnose the problem. PayFast payment notifications also log the booking's unique ID.
Why it matters
Server logs are streamed to operational tooling (Docker logs, Hostinger monitoring, any future log aggregator), retained for varying periods, and accessible to anyone with VPS console access. They are not designed for patient-medical-record handling. POPIA's data-minimisation principle says we should only retain what we need, in the right place, for the right people.
What to do
Replace the patient-details snapshot with a hash or the short booking reference. Move detailed failure diagnostics into the existing audit_log table (which IS designed for this) or into a dedicated secure-error endpoint that requires admin sign-in to read.
Effort
2-3 hours.
Why 2-3 hours

The code change itself is ~15 minutes per site, and there are only 2 sites. The rest of the time covers:

  • Triggering a fake handoff failure locally to confirm what gets redacted (~20 min).
  • Deciding the redacted format - hash vs short reference ID vs booking ID only (~15 min).
  • Confirming the equivalent detail is still recoverable from the audit_log table (where patient detail belongs) (~30 min).
  • Spot-checking past log retention to confirm the leak window closes once redacted (~30 min).
  • Buffer for deploy + sanity check in production logs (~30 min).

Complete3. Confirm the secret-config file isn't in git history

Resolved 2026-05-22 Swept the full git history with git log --all --full-history --oneline -- <path> against every plausible credential filename: .env, .env.local, .env.production, .env.development, credentials.json, secrets.json, both at the repo root and under booking-app/. Zero commits matched. Combined with the existing .gitignore entries on lines 50-51 (.env + .env.*), this confirms credentials have never been version-controlled. No key rotation required.
What it is
The reviewer flagged that the file holding production API keys, SMTP credentials, and payment-gateway secrets (.env.local) may have been committed to the repository at some point. If so, anyone with a copy of the repository (including from any historical fork or backup) can read those secrets even if the file is now removed.
Why it matters
Committed secrets cannot be "uncommitted" in git history without rewriting the history. If they were ever pushed, they have to be assumed leaked and rotated.
What to do
Check the git log for the file. If it ever appears: rotate every key it contains - Supabase service role key, PayFast credentials, CareFirst API key, SMTP password - remove the file from git history with the BFG cleaner, and confirm .env.local is in .gitignore. If it was never committed, no action needed.
Effort
15 minutes to check; 2-3 hours of rotation work only if the file was actually committed.
Why this range

15 minutes if the file is clean - one git log --all --full-history -- .env.local command tells you definitively.

2-3 hours worst case if the file is dirty:

  • Rotate 4 secrets - Supabase service role, PayFast credentials, CareFirst key, SMTP password. Each provider has a 10-15 min rotation flow.
  • Run BFG cleaner or git-filter-repo to scrub the file from git history (~30 min).
  • Force-push to remote and coordinate with anyone holding clones (~30 min).
  • Re-deploy with new secrets (~15 min).
  • Verify sign-in, payment notify, CareFirst handoff and email all still work after the rotation (~45 min).

Complete4. Database is missing indexes on the columns we query most

Resolved 2026-05-22 Migration 032_unit_scope_indexes.sql applied to production Supabase. What it is
Several columns that the application filters by on every page - for example, the column linking a booking to its unit - don't have a database index. The breaking point is roughly 1,000 bookings per unit. Beyond that the booking list page and unit-scoped audit queries get noticeably slow. Adding indexes is a one-line change.
What to do
Add database indexes on the join keys for bookings and user-unit assignments. Ship as migration 030.
Effort
1 hour.
Why 1 hour

  • Write migration 030 in the existing migration format (~10 min).
  • Apply locally and run EXPLAIN ANALYZE on representative queries to confirm Postgres picks up the index (~15 min).
  • Apply to staging and smoke-test the booking list + audit query paths (~15 min).
  • Apply to production - this is trivial since indexes are additive and cause no lock contention at this scale (~20 min including the verification window).

Complete6. Booking list re-loads everything after every change

Resolved 2026-05-22 All four booking-store mutation paths (createBooking, updateBooking, discardBooking, abandonBooking) now patch the local bookings array in place after the Supabase call instead of refetching the whole list. This is the most-used page in the application. As volume grows, the UX degrades visibly - every click feels a bit slower than the last. Also wastes database load.
What to do
Patch the local list in place after a successful update instead of refetching. Standard React pattern; ~30 lines of change in the booking store.
Effort
2-3 hours plus testing.
Why 2-3 hours

The code change is small but verification is the heavy bit:

  • Update 4 mutation points in booking-store.tsx to patch local state instead of triggering a refetch (~1 hr code).
  • Manually test each of the 4 paths - create, edit, discard, abandon - verifying scroll position is preserved and no flicker (~1 hr).

Complete7. Authentication check is duplicated across every protected endpoint

Resolved 2026-05-21 Added a default-deny check to src/middleware.ts: any /api/admin/* request without a Supabase auth cookie now gets a 401 before reaching the handler. The existing in-handler requireSystemAdminWithCaller() guards stay in place as defence-in-depth. Belt + braces: even if a future admin endpoint forgets the role check, unauthenticated traffic is blocked. Coverage enforced by a new Playwright test (tests/admin-auth-coverage.spec.ts) that walks every route.ts under src/app/api/admin/ and asserts each rejects unauthenticated requests with 401/403/405. All 23 endpoints currently pass.
What it is
Every API endpoint that requires sign-in re-runs the same 20 lines of auth-checking code. There are roughly 30 endpoints. If a new one gets added and the developer forgets to copy-paste the check, that endpoint silently has no security.
Why it matters
This is how security holes appear. It's not a current vulnerability - every existing endpoint has been checked - but the pattern is fragile. The risk grows every time a new endpoint is added.
What to do
Move the auth check into a single piece of middleware that runs for any URL under /api/admin/. Then individual endpoints only have to declare "I need admin" once.
Effort
1 day including testing every endpoint after the change.
Why 1 day

The change itself is small but the coverage matrix is large:

  • Write the middleware that runs on the /api/admin/* matcher with role-based checks (~2 hr).
  • Remove the redundant auth blocks from ~30 endpoints (~1 hr - mechanical).
  • Re-test all 30 endpoints. Each one needs three checks: signed-out returns 401, wrong role returns 403, correct role returns 200. Don't skip - auth was previously per-endpoint so any regression is silent (~5 hr).

The day reflects the testing matrix, not the typing.

Complete8. Booking state transitions are scattered across many files

Resolved 2026-05-21 New src/lib/booking-state-machine.ts defines the canonical model: a TRANSITIONS map of allowed target statuses per source, plus canTransition() / assertTransition() validators and a transitionStatus(client, id, from, to, extras) helper that combines validation with a conditional UPDATE. Five server-side transition sites migrated: PayFast ITN webhook, PayFast reconcile sweep, admin complete-payment, mark-monthly-invoice, mark-self-collect, and Start Consult (Payment Complete → Successful). Each now filters its UPDATE by the expected current status, so concurrent writers (e.g. ITN arriving mid-reconcile) settle cleanly at the DB level — the loser gets a "conflict" result instead of double-writing the row + audit log.

Follow-up (also 2026-05-21): a race-test on Start Consult revealed that while the DB transition was protected, both racing requests still called CareFirst (which is not idempotent on uniqueReference), creating duplicate SSO sessions. Added a handoff lock pattern: a conditional UPDATE acquires handoff_status = "in_progress" before the CareFirst call, returning 409 to the loser. Only one window reaches the external API.
What it is
A booking moves through statuses (In Progress to Payment Complete to Successful or Discarded/Abandoned). The rules for which transitions are allowed live in eight different files. There's no single place that says "from status X you can only go to Y or Z".
Why it matters
Two things can go wrong simultaneously - for example, manual payment confirmation by an admin happening at the same moment as automatic PayFast reconciliation - and the system has no central guard to prevent inconsistent states. We haven't been bitten by this yet, but the longer the system runs the more likely it gets.
What to do
Define the state machine in one shared module. Every transition path calls into that module rather than implementing its own checks.
Effort
1-2 days; this is a refactor, not a bug fix.
Why 1-2 days

The risk profile is the driver, not the keystroke count:

  • Map all transition logic currently scattered across booking-store.tsx and 8 API routes (~2 hr).
  • Define the state machine in one module - states + the allowed transitions between them (~3 hr).
  • Replace inline checks with calls to the shared module (~3 hr).
  • Race-condition testing - this is the big chunk. Admin manual-confirm racing with PayFast auto-reconcile, abandoned-flow racing with continue, payment-failed racing with success. Each race needs reasoning + reproduction (~4-8 hr).

It's marked 1-2 days because the lower bound assumes no surprising race interactions and the upper bound assumes one or two pieces of logic that turn out to depend on timing in subtle ways.

Half a day is the conversation + decision + write-up combined. The code is the small part.

09 - Verify How to verify any finding

This audit was conducted by reading the codebase and the dev server logs directly. To verify any specific claim:

Security claims
Open the referenced file at the referenced line. The behaviour is in the code.
Database claims
In the Supabase SQL editor, run SELECT indexname FROM pg_indexes WHERE tablename = 'bookings' and compare against the columns the app filters on.
Performance claims
Server response times are visible in the dev log. Endpoints over 1 second consistently are worth investigating.
Dead-code claims
Each cited file path is a real location in the repo. Grep for the symbol; if it has zero call sites, the claim stands.

No claim in this document was made without a verified code reference. The reviewers were instructed to flag only what they could verify, not speculate.

10 - Scope What this audit deliberately did not cover

For honesty about scope, these were not in this audit:

  • Penetration testing. This is a code review, not a live attack against the running system. Some vulnerabilities only surface under active probing.
  • Third-party dependency CVE scan. Worth running npm audit separately, ideally as part of CI.
  • Load testing. The "scalability ceiling" numbers above are estimates based on query patterns, not measured under real load.
  • Accessibility audit. WCAG conformance was not assessed.
  • Browser compatibility. Tested in development only; real-world device coverage unknown.

These are all reasonable follow-up audits depending on how the company prioritises risk.
